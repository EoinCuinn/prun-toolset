# Landing Distance — Findings and Open Questions

*Capture study of take-off/landing distance, shared to the flight-discussion channel — 2026-07-16, by Eoin Cuinn (EC3)*

This writes up what our captures show about the take-off/landing **landing distance** — the last random-looking term in the flight model — so the numbers are on the table. It is built on the full local capture corpus (80 unique `.jsonl` files, 43,149 message frames) plus a static read of the game bundle, and it confirms Marcus Licinius Crassus's landing PRNG law independently on our own data, then adds where the seed comes from. Almost nothing here is a new discovery — the law and the distance formula are prior community work — but the seed provenance and the "is it steerable" question are newly pinned down. Numbers are quoted from the source files named alongside them; the verification scripts are in `prun-flight-capture/landing-rng/`.

One convention up front: in the captured flight payloads, `segment.stlDistance` is in kilometres, and landing distance `d` below is that value for the `LANDING` segment. `k` is the per-planet constant in the law; `r` is the pseudo-random draw.

---

## Headline

Landing distance is not physics — it is a single deterministic pseudo-random draw, and the generator behind it is now fully characterised. Marcus's law `d = k(planet)·(13 + 4·r)`, with `r = new java.util.Random(UUID.hashCode(missionId)).nextDouble()`, reproduces our captured landings to machine precision. The band is `d ∈ [13k, 17k)`: expected value `15k`, spread `±2/15` (±13.3%) exactly — the "±15%" everyone has seen is that ratio. The seed is the `missionId`, a UUID the **client** generates and the server echoes verbatim (860/860 in our captures). Real committed flights inherit their plan's `missionId` and run the same draw — 21 of 25 real landings carry a byte-exact preview distance — and the **fixed-tile hypothesis is refuted**: the same ship landing at the same base draws a different distance every time. And a disassembly of the game bundle shows the client mints that UUID with `crypto.getRandomValues()` — an OS CSPRNG — so the draw is deterministic and readable but **not steerable at source**. The read-it-don't-seed-it model: you can predict a landing from the wire once the client sends the `missionId`, but you cannot make it pick a good one.

## 1. The distance formula and its mean (prior work)

The centre of the distribution — the expected landing distance — is Taiyi Bureau's distance work, driven by the destination planet's radius and pressure. The planner uses the frame-distance form `d_LDG = 926 × (R_m/1e6)^0.96 × (1+P)^(-0.42)` for the mean, which is the expected value `15k`. Taiyi's universal constant `C = 20/3` was confirmed across Promitor, Beanironia and Odysseus (45 samples). This part is unchanged; everything below is about the ±13.3% band around that mean, not the mean itself.

---

## 2. The PRNG law — seed, formula, band

The per-flight landing distance is:

```
seed  =  UUID.hashCode(missionId)                    // Java 128 → 32-bit fold
r     =  new java.util.Random(seed).nextDouble()     // Java stock LCG, first draw
d     =  k(planet) · (13 + 4·r)                       // k = one constant per planet
```

`r ∈ [0,1)` gives `d ∈ [13k, 17k)`: `dmin = 13k`, `dmax = 17k`, mean `15k`, half-width `2/15 = ±13.3%` exactly.

We ported `UUID.hashCode` (the 128→32-bit fold) and `java.util.Random.nextDouble()` to a 48-bit BigInt LCG (`rng_test.js`) and ran it against every captured `LANDING` segment:

- **`k = d/(13+4r)` is constant per planet** across different missionIds: 18 planets, worst within-planet coefficient of variation **0.0000%**.
- **Parameter-free check** (no `k` needed): the observed ratio `d₁/d₂` equals the predicted `(13+4r₁)/(13+4r₂)` on 27 planet/mission pairs, max error **0.0000%**.

This is Marcus's law, reproduced independently. His own validation is **R² = 1.000000 across 4,482 planets and ~86k flights**; ours is machine-precision agreement on the subset we hold.

## 3. The seed is client-controlled — 860/860, and real flights inherit it

The interesting part is *where the entropy enters*. The `missionId` is generated by the client and sent in the outgoing `SHIP_FLIGHT_CALCULATE_TEST_FLIGHT` request; the server echoes it verbatim and derives the landing from it.

- **Join requests to responses by `actionId`** (`seed_origin.js`): the client's `missionId` equals the server's `missionId` in **860 of 860** joined pairs. The client picks the UUID; the server trusts it.
- **Real flights** (`SHIP_FLIGHT_FLIGHT`) drop the `missionId` from their own frame and expose the destination only down to `SYSTEM/PLANET` (no finer tile). Of **25** distinct real landings, **21** carry a landing distance byte-identical to a captured preview on the same planet (`seed_crack.js`, `seed_verify.js`) — 15-significant-digit float agreement, which is not chance: those flights ran their plan's draw rather than re-rolling.

One honesty note on the metric: `seed_verify.js` re-derives `r` from the twin and reports `0.00000%` error, but that is **by construction** — the twin is *selected* by `twin.d == f.d`, and `k·(13+4r)` already reproduces a mission's own distance to machine precision, so the "prediction" re-derives the join key and cannot fail. The real evidence is the *existence* of a byte-exact twin, not a 0.00000% error. The **4** real flights without a twin all fall inside `[13k, 17k)` for their planet (`d/k` = 14.37, 15.55, 15.65, 16.31, on planets with 67–186 captured missions) — ordinary draws whose preview was not captured, not outliers.

(The dead end that pointed here: the real flight id, tried as the seed under three hashings, reproduced `r` on 0/13 flights — `seed_hunt.js`. The flight id is not the seed; the plan's `missionId` is.)

## 4. The bundle — CSPRNG, so unsteerable at source

The obvious next thought is: if the seed is a client-chosen UUID and `r = f(missionId)` computes offline, can you choose a favourable `missionId`? A static read of the game bundle (`bundle.1ac60af678af4c00.js`) closes this.

The dispatch site (offset 4,797,518) passes `missionId` in as an argument; the generator `Wo()` (offset 1,978,919) is a textbook UUID v4 — 16 bytes, version/variant bits set — fed by `crypto.getRandomValues()`, the OS CSPRNG entropy pool. The bundled `uuid` module has an `e.random`/`e.rng` override seam, but it is never exercised — every call is `Wo()` with no args. `Math.random` appears 60 times in the bundle and 0 times in the `uuid` module. No code path seeds the UUID.

So: the draw is deterministic *given* the `missionId`, and the `missionId` is client-supplied and server-trusted — meaning it is selectable *in principle*. But the stock client mints it from cryptographic entropy, so a player cannot make it land on a chosen value. The only way to steer it would be to forge the outgoing `missionId` before it is sent (which the 860/860 echo shows the server would honour) — i.e. active frame injection, not normal play. **This documents the mechanism; it is not a how-to.** For a read-only tool the operating model is: expected value `15k` when the `missionId` is unseen, exact when it is read from a live capture. Selectability is a theoretical property of the mechanic, not a feature.

## 5. Fixed-tile hypothesis — refuted

The open question from the prior work was whether real flights *to an established base* land on a fixed tile instead of re-rolling. The corpus refutes it. Grouping the 25 real landings by `(shipId, planet)` gives six cases of the same ship landing at the same planet two or more times; a company holds at most one base per planet and these are all to planets where the company has a base, so same ship + same planet is the same base. Every one landed at a different distance:

| Planet | shipId | Landing distances (km) | Spread |
|--------|--------|------------------------|--------|
| VH-331g | `58d4a04c66` | 4336.3 → 5118.3 | 18.0% |
| VH-331g | `2e44055218` | 4368.9 → 4826.4 | 10.5% |
| OF-443b | `58d4a04c66` | 8279.2 → 8970.5 | 8.3% |
| VH-331a | `58d4a04c66` | 4426.5 → 4750.4 | 7.3% |
| OF-443b | `59b1a30705` | 7725.9 → 7905.0 | 2.3% |
| OZ-189a | `2e44055218` | 3654.2 → 3712.1 | 1.6% |

At the planet level this is `seed_crack.js` TEST 2: **6 destinations flown ≥2 times, 0 with a constant distance, all 6 varying** (VH-331g: 7 flights, 7 distinct distances, 4040.9–5118.3 km). A fixed base tile is a property of the base, so it would be ship-independent and give one constant distance per base — instead every base re-rolls, and since landing distance is deterministic to machine precision, even the 1.6% (58 km) gap is a genuinely different draw. This test uses same-base repeats, not the byte-exact-twin join of §3, so it is free of that join's selection bias. Real flights to an established base draw the same PRNG as previews; there is no fixed tile.
